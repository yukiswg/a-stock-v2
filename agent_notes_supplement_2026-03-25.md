# 第一轮：Agent 基础

## Q1
### 题目
Agent 和普通 prompt 的本质区别是什么？

### 标准答案
Prompt 是 `single-pass generation`，一次前向生成直接输出结果；  
Agent 是一个基于 loop 的决策系统，模型在每一步都会根据当前 state 做决策，选择 action，并更新状态，直到任务完成。

### 你的答案
“agent 是一个决策的 while 循环，而 prompt 只是一次单词的命令。”

### 严格点评
- 方向是对的，但表达很差。
- “一次单词的命令”这种说法太口语，也太弱，面试官会直接觉得你没抽象能力。
- 你说出了 `while`，但没有说出 `single-pass generation`，这说明你知道现象，不会标准定义。
- 这类题不能靠口语对冲，必须上术语。

### 你这里暴露的问题
- 定义不够精确
- 缺少术语级表达
- 回答像聊天，不像面试

## Q2
### 题目
Agent 的 loop 里最关键的 3 个元素是什么？

### 标准答案
Agent loop 的核心三要素是：
- `State`
- `Policy`
- `Action`

### 你的答案
“判断是否完成命令，执行，更新状态。”

### 严格点评
- 这是流程描述，不是机制定义。
- 你漏掉了最关键的 `Policy`，这会让你的回答直接降级。
- 面试官要听的是组件抽象，不是你口头复述执行顺序。
- “判断是否完成”最多算终止条件，不是 Agent 核心三要素。

### 你这里暴露的问题
- 把“步骤”当成“结构”
- 没有抓住决策模块这个核心
- 缺少对系统抽象层次的意识

## Q3
### 题目
为什么说 Agent 本质更接近强化学习（RL）里的范式？

### 标准答案
Agent 和 RL 相似，是因为两者本质上都是 `sequential decision making` 问题：
- 都有 state
- 都有 policy
- 都在多步过程中优化决策路径

区别在于：
- RL 显式优化 reward
- Agent 往往通过 prompt、heuristic、feedback 等隐式方式优化行为

### 你的答案
“强化学习里主要是决策，然后看分布是否离正确答案更近，近就奖励，远就惩罚；agent 也是决策、执行、更新状态。”

### 严格点评
- 这段回答非常危险。
- 你把 RL 说成“接近答案就奖励”，这不是机制描述，而是非常初级的表面化理解。
- 你没有说 `policy optimization`，也没有明确说 `sequential decision making`。
- 你的回答听起来像知道 RL 的皮毛，但没有进入 formal view。

### 你这里暴露的问题
- 对 RL 的理解还停留在“奖励/惩罚”的口头层面

# 第二轮：Memory / Workflow / Stability

## Q1
### 题目
如果没有 memory，Agent 会发生什么问题？

### 标准答案
没有 memory 的本质问题是：Agent 无法进行 `state accumulation`。  
具体表现：
- 每一步都只能做局部最优决策
- 无法利用历史结果
- 容易重复调用工具，出现 loop 或 oscillation

### 你的答案
“没有 memory，agent 可能无法进一步接近正确答案，会一直卡在初始阶段，无法更新状态，可能会不断询问。”

### 严格点评
- 你答得太泛。
- “无法接近正确答案”是结果，不是机制。
- 真正该说的是“无法跨步累积信息”，这是 memory 的本质作用。
- 你用了“卡住”“不断询问”这种现象描述，但没有说明为什么会这样。

### 你这里暴露的问题
- 只会讲后果，不会讲机制
- 抓不住 memory 的功能本质
- 回答缺乏技术化表达

## Q2
### 题目
Agent 和 workflow（流程编排）的本质区别是什么？

### 标准答案
一句话版本：  
`Workflow` 是 `static graph`，`Agent` 是 `dynamic policy`。

展开：
- Workflow 的路径是预先定义好的
- Agent 的路径由模型在 runtime 根据 state 动态决定

### 你的答案
“workflow 没有一个回头去看是否满足需求的过程；agent 如果发现不满足，就会继续执行之前的步骤，直到条件合理。”

### 严格点评
- 这个回答太表面。
- “会不会回头看”不是本质区别。
- 面试官要听的是：路径是否固定，决策是否在线生成。
- 你说的是行为现象，不是结构差异。

### 你这里暴露的问题
- 不会把相似系统拉开层级差异
- 习惯从使用体验角度回答，而不是从架构角度回答
- 缺少 `static graph / dynamic policy` 这种一锤定音的表述

## Q3
### 题目
为什么很多 Agent 实际上不稳定（unstable）？

### 标准答案
典型原因有三类：
- `Policy non-determinism`：同一 state 可能产生不同 action
- `Error propagation`：中间步骤一旦出错，后面会被连锁带偏
- `Reward / evaluation ambiguity`：缺少明确成功信号，不知道何时停、何时算完成

### 你的答案
“判断现在是否满足标准写得模糊，所以答案会稳定或者不稳定；还有问题是否定义清楚。”

### 严格点评
- 这类回答太像产品经理，不像算法候选人。
- 你完全没落到 `policy`、`state evolution`、`error propagation` 这些机制词上。
- “标准模糊”只能算一个表层原因，不能代替结构性解释。
- 如果面试官追问“为什么模糊会导致不稳定”，你大概率接不住。

### 你这里暴露的问题
- 会说业务现象，不会说模型机制
- 缺少 error propagation 这种系统视角
- 没把不稳定性拆成可分析的来源

# 第三轮：Tool Use / RAG / 死循环

核心价值：
- 把问题转化为 API / code / DB 等可执行操作
- 用确定性执行结果替代不可靠的 token prediction
- 把复杂任务拆成“决策 + 执行”两部分

- 真正本质是：LLM 从 token predictor 变成 execution system。
- 你没有把“信息获取”和“可执行能力”区分开，所以层次不够。

## Q3
### 题目
如果 Agent 一直调用 tool 不停（死循环），你怎么解决？

- `Loop detection / state diff`：检测重复 tool 调用或 state 无变化时强制终止

## Q1
### 题目
Reward Model 的输入和输出分别是什么？

### 标准答案
Reward Model 的输入通常是：
- 一个 prompt
- 以及对应的候选回答（单个回答打分，或成对回答做偏好比较）

Reward Model 的输出是：
- 一个标量 reward score，表示这个回答有多符合人类偏好

更严谨一点：
- 训练时经常使用“优选回答 vs 差回答”的 pairwise preference 数据
- 推理时本质上是在给某个回答打一个偏好分数

### 严格提醒

- 它不是生成答案的模型，而是评估回答偏好分数的模型。

## Q2
### 题目
为什么不能只用 SFT，而要引入 RLHF？

### 标准答案
因为 SFT 本质是模仿参考答案分布，它能让模型学会“像样地回答”，但不擅长优化细粒度的人类偏好。  
引入 RLHF 的原因是：
- 很多问题没有唯一标准答案
- 人类更关心“哪个回答更好”，而不只是“像不像训练样本”
- RLHF 可以借助 Reward Model 直接优化输出偏好，而不是只做监督模仿

一句话版：  
`SFT 学 imitation，RLHF 学 preference optimization。`

## Q3
### 题目
PPO 在 RLHF 里到底优化的是什么？

### 标准答案
PPO 在 RLHF 里优化的不是“效果”这种模糊对象，而是语言模型的 `policy`。  
更具体地说，它优化的是：
- 模型生成分布，也就是 `pi_theta(y | x)`
- 目标是在 reward 更高的方向上更新 policy
- 同时通常会配合 KL 约束，避免模型偏离初始参考策略太远

一句话版：  
`PPO 优化的是 LLM 的生成策略（policy），不是在空泛地“调效果”。`
